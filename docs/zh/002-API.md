# API

API 使用 options-first 风格，并优先描述 GitHub Actions 概念，再描述 backend 实现细节。`act` 目前是默认 engine，但 engine 选项位于明确的 engine 边界之后。

## GitHub Facade

```ts
import { inject, InjectionToken } from "injection-js";
import {
	createCheckoutPretender,
	createRemoteMockPretenders,
} from "pretend-act/actions";
import { ActEngine, AgentCiEngine } from "pretend-act/engine";
import {
	createGithubActionsContainer,
	createRemoteMockContainer,
	GithubActionsWorkspaceToken,
} from "pretend-act/github";
import { PretendRunner } from "pretend-act/runner";
```

`createGithubActionsContainer(options)` 是 GitHub Actions simulation 的显式依赖 resolver。它会创建一个由 `injection-js` injector 支撑的 disposable container。内置 provider 包括 `GithubActionsWorkspaceToken` 和 `GithubCheckoutBackendToken`；自定义 pretender 可以通过 `options.providers` 增加 provider，并用 `inject(Token)` 读取。可选依赖用 `container.get(token)`，必须存在的依赖用 `container.require(token)` 进行快速失败。

`PretendRunner` 是主要编排入口。它接收 engine adapter、显式 `cwd`、workflow overlay、action pretender 和可选 `injector`。Pretender matcher 和 pretender 会在这个 injection context 中运行，因此可以在构造 replacement operation 时同步调用 `inject()`。它提供 options-first 的运行方法：

- `validateWorkflow(options?)`
- `runWorkflow(options?)`
- `dryRunWorkflow(event, options?)`
- `runEvent(event, options?)`
- `runJob(jobId, options?)`

`runWorkflow()` 是 workflow-first 入口，用于 `AgentCiEngine` 这类不把本地执行建模为指定 event command 的 backend。`AgentCiEngine` 是可选集成。默认会先解析 optional peer `@redwoodjs/agent-ci` 的 package bin，再回退到 `PATH` 中的 `agent-ci`；`agentCiBinary` 或 `AGENT_CI_BINARY` 可以指定自定义 binary。binary 缺失或不可执行时会返回 Agent CI 专属诊断。

Engine 会声明 capabilities。`PretendRunner` 会在 staging workflow 或启动 backend 前检查这些 capabilities，因此 `AgentCiEngine` 搭配 `runEvent()` 这类不支持组合会提前得到 engine-aware 错误。

自定义 pretender 使用标准 `injection-js` token，而不是 decorator：

```ts
const ReleaseToolToken = new InjectionToken<{ command: string }>("release tool");

await using container = await createGithubActionsContainer({
	providers: [{ provide: ReleaseToolToken, useValue: { command: "pnpm pack" } }],
	repository: { name: "repo", source: { path: process.cwd() } },
});

const runner = new PretendRunner({
	cwd: container.require(GithubActionsWorkspaceToken).repoPath,
	engine: new ActEngine(),
	injector: container.injector,
	actions: {
		pack: {
			test: ({ step }) => step.run === "npm publish",
			pretender: () => ({ operation: "replace", with: inject(ReleaseToolToken).command }),
		},
	},
});
```

Container 实现 `[Symbol.asyncDispose]`，因此推荐用 `await using` 表示生命周期边界。只需要复制 source 的测试可以传入 `repository.checkout: false`。

```ts
const runner = new PretendRunner({
	cwd: process.cwd(),
	engine: new AgentCiEngine({ quiet: true, agentCiStateDir: ".agent-ci-state" }),
	workflowFile: ".github/workflows/ci.yml",
});

await runner.runWorkflow();
```

## Configuration

```ts
import { defineConfig, PretendEngineType } from "pretend-act/config";
```

`defineConfig(config)` 为 `pretend-act.config.ts` 和程序式 setup 保留类型信息。顶层结构是 `engine`、`actions` 和 `runner`：engine 专属字段放在 `engine.options`，action pretender 注册在 `actions`，GitHub repository/runtime 模型放在 `runner.github`。

## Git Registry Checkout

```ts
import { createCheckoutReplacementStep, createGitRegistry, startGitHttpTransport } from "pretend-act/git-registry";
```

`createGitRegistry(options)` 会创建一个用于 checkout simulation 的本地 Git registry。auto mode 下，脏工作区会变成 filtered snapshot commit；显式 `sourceSha`、tag 和 clean `sourceRef` branch 会直接发布选定 commit。返回对象包含 `remoteUrl`、`transport`、`fetchRef`、`checkoutSha`、`mode`、`stop()`，以及供 `await using` 使用的 `[Symbol.asyncDispose]()`。默认 transport 返回本地 `file:` URL；当 runner 需要通过 HTTP fetch 时设置 `transport: "http"`。`startGitHttpTransport(options)` 也可以暴露已有 registry directory。

`createCheckoutReplacementStep(server)` 返回内置 checkout pretender 使用的 shell step。高层调用方通常把 `container.injector` 传给 `PretendRunner`，再使用 `createCheckoutPretender()`；该 pretender 会通过 `inject()` 读取 `GithubCheckoutBackendToken`。

## Workflow Overlay

```ts
workflowOverlay: [
	{
		type: "skip-step",
		jobId: "release-plan",
		selector: { uses: "actions/upload-artifact@v7" },
	},
]
```

Step 可以按 `id`、`name`、`uses`、`run` 或 `index` 匹配。支持 `replace-step`、`skip-step`、`insert-step-before`、`insert-step-after` 和 `keep-step`。Overlay 只写入 sandbox workflow 文件。Action pretender 会把 action-level rule 编译成同一个 overlay model。

## Result

```ts
type RunResult = {
	command: CommandSpec;
	status: "success" | "failure" | "cancelled" | "unknown";
	success: boolean;
	exitCode: number | null;
	stdout: string;
	stderr: string;
	rawLog: string;
	jobs: JobResult[];
};
```

Raw log 和 exit code 是可靠来源。Step 级 summary 保守处理，无法可靠推导时会是 `unknown`。`AgentCiEngine` 会从 `AGENT_CI_STATE_DIR` 读取 Agent CI 的 run-result JSON，并把可用 job/step summary 映射到 `jobs`；如果 backend 没有产出结果文件，`jobs` 为空。

## Registry Helpers

`pretend-act/npm-registry` 提供配置片段和 Verdaccio lifecycle helper。`pretend-act/crates-registry` 提供 Cargo config，以及具备核心 Web API 的 sparse registry server。`pretend-act/docker-registry` 使用 Dockerode 启动和停止官方 OCI Distribution registry container。`pretend-act/github` 暴露 `createRemoteMockContainer` 编排这些服务，并默认 deny 真实发布端点。

`pretend-act/actions` 暴露 `createRemoteMockPretenders(remoteMock, options?)`，可以把 remote mock container 转成 action pretender，以及接入 runner 所需的 `env`、`secrets` 和 `providers`：

```ts
await using remoteMock = await createRemoteMockContainer({ npm: true, crates: true });
const remoteMockPretenders = createRemoteMockPretenders(remoteMock);

const runner = new PretendRunner({
	cwd: process.cwd(),
	engine: new ActEngine(),
	actions: remoteMockPretenders.actions,
});

await runner.runEvent("workflow_dispatch", {
	env: remoteMockPretenders.env,
	secrets: remoteMockPretenders.secrets,
});
```

内置 npm 和 Cargo pretender 有意保持保守：默认只匹配简单的 `npm publish`、`pnpm publish` 和 `cargo publish` run step，写入本地 registry 配置，并 keep 已经显式指定 registry override 的 step。Docker publish rewrite 只能通过 `createDockerPublishPretender()` 或 `createRemoteMockPretenders(..., { docker: ... })` 显式 opt in；它不会解析任意 shell script，也不承诺 GHCR/Docker Hub auth parity。
