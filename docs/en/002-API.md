# API

The API is options-first and describes GitHub Actions concepts before backend implementation details. `act` is currently the default engine, but engine options stay behind explicit engine boundaries.

## GitHub Facade

```ts
import { inject, InjectionToken } from "injection-js";
import {
	createCheckoutPretender,
	createRemoteMockPretenders,
} from "pretend-act/actions";
import { ActEngine, AgentCiEngine } from "pretend-act/engine";
import {
	createGithubActionsContainer,
	createRemoteMockContainer,
	GithubActionsWorkspaceToken,
} from "pretend-act/github";
import { PretendRunner } from "pretend-act/runner";
```

`createGithubActionsContainer(options)` is the explicit dependency resolver for GitHub Actions simulations. It creates a disposable container backed by an `injection-js` injector. Built-in providers include `GithubActionsWorkspaceToken` and `GithubCheckoutBackendToken`; custom pretenders can add more providers through `options.providers` and read them with `inject(Token)`. Use `container.get(token)` for optional providers and `container.require(token)` when a missing provider should fail fast.

`PretendRunner` is the primary orchestration entry. It accepts an engine adapter, an explicit `cwd`, workflow overlays, action pretenders, and an optional `injector`. Pretender matchers and pretenders run inside that injection context, so `inject()` can be called synchronously while building a replacement operation. It exposes options-first run methods:

- `validateWorkflow(options?)`
- `runWorkflow(options?)`
- `dryRunWorkflow(event, options?)`
- `runEvent(event, options?)`
- `runJob(jobId, options?)`

`runWorkflow()` is the workflow-first entrypoint for backends such as `AgentCiEngine` that do not model local execution as an event-specific command. `AgentCiEngine` is optional. By default it resolves the package bin from optional peer `@redwoodjs/agent-ci`, then falls back to `agent-ci` on `PATH`; `agentCiBinary` or `AGENT_CI_BINARY` can point to a custom binary. Missing or non-executable binaries fail with Agent CI-specific diagnostics.

Engines declare capabilities. `PretendRunner` checks those capabilities before staging workflows or spawning the backend, so unsupported combinations such as `AgentCiEngine` with `runEvent()` fail early with an engine-aware error.

Custom pretenders use normal `injection-js` tokens rather than decorators:

```ts
const ReleaseToolToken = new InjectionToken<{ command: string }>("release tool");

await using container = await createGithubActionsContainer({
	providers: [{ provide: ReleaseToolToken, useValue: { command: "pnpm pack" } }],
	repository: { name: "repo", source: { path: process.cwd() } },
});

const runner = new PretendRunner({
	cwd: container.require(GithubActionsWorkspaceToken).repoPath,
	engine: new ActEngine(),
	injector: container.injector,
	actions: {
		pack: {
			test: ({ step }) => step.run === "npm publish",
			pretender: () => ({ operation: "replace", with: inject(ReleaseToolToken).command }),
		},
	},
});
```

Containers implement `[Symbol.asyncDispose]`, so `await using` is the preferred lifecycle boundary. Pass `repository.checkout: false` when a test only needs the copied source.

```ts
const runner = new PretendRunner({
	cwd: process.cwd(),
	engine: new AgentCiEngine({ quiet: true, agentCiStateDir: ".agent-ci-state" }),
	workflowFile: ".github/workflows/ci.yml",
});

await runner.runWorkflow();
```

## Configuration

```ts
import { defineConfig, PretendEngineType } from "pretend-act/config";
```

`defineConfig(config)` preserves typed configuration for `pretend-act.config.ts` and programmatic setup. The top-level shape is `engine`, `actions`, and `runner`: engine-specific fields stay under `engine.options`, action pretenders are registered under `actions`, and the GitHub repository/runtime model lives under `runner.github`.

## Git Registry Checkout

```ts
import { createCheckoutReplacementStep, createGitRegistry, startGitHttpTransport } from "pretend-act/git-registry";
```

`createGitRegistry(options)` creates a local Git registry for checkout simulation. In auto mode, dirty workspaces become filtered snapshot commits; explicit `sourceSha`, tags, and clean `sourceRef` branches publish the selected commit directly. The returned object includes `remoteUrl`, `transport`, `fetchRef`, `checkoutSha`, `mode`, `stop()`, and `[Symbol.asyncDispose]()` for `await using`. The default transport returns a local `file:` URL; set `transport: "http"` when the runner needs to fetch over HTTP. `startGitHttpTransport(options)` can also expose an existing registry directory.

`createCheckoutReplacementStep(server)` returns the shell step used by the built-in checkout pretender. High-level callers usually pass `container.injector` to `PretendRunner` and use `createCheckoutPretender()`, which reads `GithubCheckoutBackendToken` with `inject()`.

## Workflow Overlay

```ts
workflowOverlay: [
	{
		type: "skip-step",
		jobId: "release-plan",
		selector: { uses: "actions/upload-artifact@v7" },
	},
]
```

Steps can be selected by `id`, `name`, `uses`, `run`, or `index`. Supported operations include `replace-step`, `skip-step`, `insert-step-before`, `insert-step-after`, and `keep-step`. The overlay is written only to sandbox workflow files. Action pretenders compile action-level rules into the same overlay model.

## Result

```ts
type RunResult = {
	command: CommandSpec;
	status: "success" | "failure" | "cancelled" | "unknown";
	success: boolean;
	exitCode: number | null;
	stdout: string;
	stderr: string;
	rawLog: string;
	jobs: JobResult[];
};
```

Raw log and exit code are the reliable source. Step-level summaries are conservative and may be `unknown`. `AgentCiEngine` reads Agent CI's run-result JSON from `AGENT_CI_STATE_DIR` and maps available job/step summaries into `jobs`; when no result file is produced, `jobs` is empty.

## Registry Helpers

`pretend-act/npm-registry` provides configuration fragments and Verdaccio lifecycle helpers. `pretend-act/crates-registry` provides Cargo config plus a sparse registry server with core Web API support. `pretend-act/docker-registry` uses Dockerode to start and stop an official OCI Distribution registry container. `pretend-act/github` exposes `createRemoteMockContainer` to orchestrate these services and deny real publishing endpoints by default.

`pretend-act/actions` exposes `createRemoteMockPretenders(remoteMock, options?)`, which turns a remote mock container into action pretenders plus the `env`, `secrets`, and `providers` needed to wire them into a runner:

```ts
await using remoteMock = await createRemoteMockContainer({ npm: true, crates: true });
const remoteMockPretenders = createRemoteMockPretenders(remoteMock);

const runner = new PretendRunner({
	cwd: process.cwd(),
	engine: new ActEngine(),
	actions: remoteMockPretenders.actions,
});

await runner.runEvent("workflow_dispatch", {
	env: remoteMockPretenders.env,
	secrets: remoteMockPretenders.secrets,
});
```

The built-in npm and Cargo pretenders are intentionally conservative: they match simple `npm publish`, `pnpm publish`, and `cargo publish` run steps, write local registry config, and keep steps that already provide explicit registry overrides. Docker publish rewriting is opt-in only through `createDockerPublishPretender()` or `createRemoteMockPretenders(..., { docker: ... })`; it does not parse arbitrary shell scripts or claim GHCR/Docker Hub auth parity.
